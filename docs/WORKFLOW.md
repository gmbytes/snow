## 一、代码开发

### 1. 新增功能

**触发条件**：需要为项目添加新能力

**流程步骤**：

```
1. 确认需求范围
   └─ 明确功能边界、接口契约

2. 选择模块归属
   └─ 公共能力 → pkg/
   └─ 内部实现 → internal/
   └─ 参考：ARCHITECTURE.md

3. 编写测试
   └─ 面向外部行为，避免实现细节断言
   └─ 参考：TESTING.md

4. 实现代码
   └─ 遵守依赖方向：pkg/* 不依赖 internal/*

5. 本地验证
   └─ make test 或 go test ./...
```

**输出**：功能代码 + 测试代码

**强制约束**：

- `pkg/*` → 不依赖 → `internal/*`
- `routine/*` 可依赖 `pkg/*` 和 `internal/*`
- 外部项目（如 server）只能依赖 `pkg/*` 和 `routine/*`

---

### 2. 修复 Bug

**触发条件**：发现代码缺陷或用户报告问题

**流程步骤**：

```
1. 复现问题
   └─ 记录复现步骤、环境信息

2. 定位根因
   └─ 追踪调用链，确认问题位置

3. 编写回归测试
   └─ 先写失败测试，再修复

4. 修复代码
   └─ 最小改动原则，避免引入新问题

5. 验证修复
   └─ 本地测试 + 确认原问题不再复现
```

**输出**：修复代码 + 回归测试

---

### 3. 代码优化

**触发条件**：性能问题、可读性问题、重复代码

**流程步骤**：

```
1. 识别优化点
   └─ 性能分析 / 代码审查发现

2. 评估影响范围
   └─ 是否影响外部行为

3. 编写基准测试（性能优化）
   └─ 记录优化前数据
   └─ 参考：BENCHMARK_BASELINE.md

4. 实施优化
   └─ 保持外部行为不变

5. 验证效果
   └─ 对比优化前后数据
```

**输出**：优化后的代码 + 基准测试（如有）

---

### 4. 重构

**触发条件**：架构违规、技术债务积累、模块职责不清

**流程步骤**：

```
1. 识别重构目标
   └─ 对照 ARCHITECTURE.md 识别违规点
   └─ 记录待办事项

2. 制定计划
   └─ 拆分为小步可验证的子任务

3. 执行重构（每轮）
   ├─ 实施改动：每轮一个最小可验证单元
   ├─ 验证通过：make test 或 go test ./...
   └─ 更新状态：同步任务清单

4. 完成验收
   └─ 确认所有子任务完成
   └─ 更新相关文档（如有接口变更）
```

**输出**：重构后的代码 + 更新的文档

**关键原则**：

- 每轮只做一个最小可验证改动
- 保持测试始终通过
- 及时同步文档状态

## 二、协作流程

### 5. 提交 MR

**触发条件**：代码开发完成，准备合并

**前置要求**：

```
├─ 本地安装 git hook：make init-hooks
└─ 钩子会在 pre-commit 时自动运行：make lint + make test
```

**流程步骤**：

```
1. 本地自检（hook 会自动执行）
   ├─ make lint 通过
   ├─ make test 通过
   └─ make test-race 通过（强烈推荐）

2. 提交代码
   └─ 创建分支，提交变更

3. 发起 MR
   ├─ 标题格式：[feat/fix/refactor] 简短描述
   ├─ 内容包含：变更说明、关联 Issue（如有）
   └─ 等待 CI 和审查
```

**输出**：Merge Request

---

### 6. 代码审查

**触发条件**：MR 提交后

**审查要点**：

```
├─ 架构边界
│  └─ pkg/* 不依赖 internal/*
│  └─ external（如 server）只能依赖 pkg/* 和 routine/*
│
├─ 代码质量
│  └─ 命名清晰
│  └─ 逻辑简洁
│  └─ 无冗余代码
│
├─ 测试覆盖
│  └─ 关键路径有测试
│  └─ 测试面向行为
│
└─ 文档同步
   └─ 接口变更有文档说明
```

**输出**：审查意见 / 批准合并

---

### 7. CI 验证

**触发条件**：MR 提交后自动触发（GitLab CI）

**验证内容**：

```
├─ make tools          （安装工具）
├─ make ci           （lint + test + test-race）
└─ 代码规范检查
```

**详细步骤**（`.gitlab-ci.yml`）：

```
stage: ci
script:
  - make tools
  - make ci
```

**结果**：通过 / 失败 → 失败则阻塞合并

## 三、发布流程

### 8. 版本发布

**触发条件**：功能就绪，准备发布

**流程步骤**：

```
1. 发布准备
   ├─ 确认 make ci 通过
   └─ 确认所有 MR 已合并

2. 打 tag
   └─ git tag v1.x.x

3. 编写发布说明
   ├─ 新增功能
   ├─ 变更内容
   └─ 兼容性说明（如有）

4. 发布
   └─ 推送 tag，触发 CD 流程
```

**输出**：版本 tag + 发布说明

---

### 9. 用户升级

**触发条件**：新版本发布后，用户升级依赖

**适用范围**：使用 `github.com/gmbyte/snow` 的下游项目（如 server）

**流程步骤**：

```
1. 阅读发布说明
   └─ 了解变更内容

2. 更新依赖
   └─ go get github.com/gmbyte/snow@v1.x.x
   └─ server 使用 replace：replace github.com/gmbyte/snow => ../snow

3. 验证
   └─ make test
   └─ 如有兼容性问题，按发布说明处理

4. 部署
   └─ 正常运行服务
```

**server 项目特别注意**：

server 项目使用 `replace` 指令指向本地 snow 目录，开发时可实时调试框架变更：

```go
replace github.com/gmbyte/snow => ../snow
```

**输出**：升级后的项目

## 四、维护流程

### 10. 问题处理

**触发条件**：用户反馈、监控告警、测试失败

**流程步骤**：

```
1. 问题记录
   ├─ 描述现象
   ├─ 复现步骤
   └─ 影响范围

2. 优先级评估
   ├─ P0：阻塞核心功能
   ├─ P1：影响重要功能
   └─ P2：次要问题

3. 处理
   ├─ P0：立即处理
   ├─ P1：排期处理
   └─ P2：酌情处理

4. 验证关闭
   └─ 确认问题解决
   └─ 补充测试防止回归
```

**输出**：问题修复 / 说明文档

## 五、强制约束

以下约束适用于所有流程，违反的 MR 不予合并：

```
pkg/* → 不依赖 → internal/*
test/integration → 不依赖 → internal/*  （如存在）
external（如 server）→ 只能依赖 → pkg/* 和 routine/*
```

## 六、本地开发环境

### Git Hook 安装

```bash
make init-hooks
```

安装后，每次 `git commit` 前会自动执行：

```bash
make lint
make test
```

如需跳过 hook 强制提交：添加 `--no-verify` 参数（不推荐）

### IDE 跨模块支持

```bash
make init-workspace
```

生成 `go.work` 使 IDE 支持 snow、examples、server 多模块联调。

## 八、两仓库关系

snow 项目包含两个独立 git 仓库：

| 仓库 | 路径 | 用途 |
|------|------|------|
| **snow** | `slgame/snow/` | 框架核心，提供依赖注入、日志、配置等基础能力 |
| **server** | `slgame/server/` | 游戏服务端实现，依赖 snow 框架 |

开发流程：

1. 修改 snow 框架 → 独立提交 → 发布新版本
2. server 通过 replace 引用本地 snow 开发版，或使用 go get 拉取发布版
3. 两仓库各自维护独立的 CI 流水线
